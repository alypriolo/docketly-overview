Docketly
Mass tort case management, built for the real world
A fully local, encrypted desktop application for managing mass tort litigation. Every byte stays on your machine. No cloud. No compromises.

The Opportunity

The leading case management tools have migrated to the cloud, creating a critical data protection gap for law firms handling confidential mass tort matters. Docketly closes that gap with a modern desktop application that keeps every file, every client record, and every privileged communication exactly where it belongs: on your machine.

Context
The Problem
Cloud-based tools introduce unacceptable risk for confidential litigation. Mass tort cases involve thousands of clients, medical records, and privileged communications. The leading solution, CaseMap+, requires all of this to live on third-party servers. Lawyers have widely reported the cloud version is slower, harder to use, and less reliable than its legacy desktop predecessor. A fast, local, privacy-first alternative has clear and unmet demand.
Approach
The Solution
Docketly runs entirely on the user's machine. All data is AES-256 encrypted at rest and nothing ever leaves the device. It brings the organizational depth of CaseMap together with a modern interface, a full mass tort lifecycle timeline with drill-down navigation, and the data protection posture that legal ethics actually requires.
Target Users
01
Solo Practitioners
Attorneys managing complex litigation independently who need enterprise-grade organization without the cost, complexity, or data exposure of enterprise tools.
02
Small Law Firms
2 to 10 attorney firms handling plaintiff-side mass tort or personal injury litigation across multiple concurrent matters.
03

Litigation teams requiring strict local data residency for regulatory compliance, client confidentiality mandates, or conflict-of-interest isolation.
Core Features / V1
F-01
Case and Client Management
Organize matters with multiple clients per case. Track diagnosis dates, medical history, opposing counsel, retained experts, and witnesses in a single structured workspace per matter.
F-02
Litigation Timeline
Full mass tort lifecycle across 7 color-coded phases from initial investigation through post-settlement. List and Gantt views, switchable. Drill down from phase to event to linked documents and individual evidence.
F-03
Document Management
Upload and encrypt medical records, briefs, depositions, correspondence, and expert reports. Integrated PDF viewer using in-memory decryption only. Bates numbering and privilege designation supported.
F-04
Facts Table
Structured fact entry linked to source documents. Strength ratings with a color-coded heat map showing what supports or undermines the case. Filterable and exportable to CSV for trial preparation.
F-05
Entity Recognition
Automatically extracts people, organizations, and key dates from uploaded documents. Cross-references entities across all case files to surface connections, patterns, and gaps in the record.
F-06
Search, Export, and Backup
Global search across all cases, documents, facts, and parties. Export timelines to PDF and facts to CSV. Full encrypted backup and restore for safeguarding the entire case database.
Security and Data Protection
Encryption
AES-256 at Rest
The entire SQLite database and every stored file are encrypted with AES-256. Physical access to the machine yields nothing without the master password. Documents are decrypted in memory only and never written to disk in plain text.
Network
Zero External Connectivity
Electron's Content Security Policy is configured to block all outbound requests with no exceptions. No analytics, no telemetry, no cloud sync. Every dependency is bundled at build time and nothing is fetched at runtime.
Authentication
Password-Derived Keys
The master password is never stored. At login, PBKDF2 derives a cryptographic key that lives in memory for the session only. A configurable idle timeout triggers auto-lock, wiping the key from memory entirely.

Technology Stack
Desktop Shell
Electron
Packages the application as a native executable for both Windows and macOS from a single codebase. More importantly, it provides access to Node.js APIs that are unavailable in a browser, which is a hard requirement for encrypting files directly on disk and holding the SQLCipher database key in memory without exposing it to the renderer process.
UI Framework
React 18
Docketly's interface is composed of many independently stateful views: a drill-down timeline, a filterable facts table, a document viewer, and an entity panel. React's component model manages this complexity cleanly while its large ecosystem ensures long-term maintainability. It is also already familiar from the portfolio project, reducing ramp-up time.
Language
TypeScript (strict mode)
Legal data structures are complex. TypeScript's static type system enforces correctness at compile time rather than at runtime, making it significantly harder to pass malformed data between the database, the IPC bridge, and the UI. For a security-critical application where data integrity is non-negotiable, this is especially valuable in preventing logic errors that could compromise case records.
Build Tooling
Vite
Provides fast hot module replacement during development and optimized production builds that bundle all dependencies into the application at compile time. Because Docketly enforces a strict zero-network policy, every asset must be self-contained at build time with no runtime fetching allowed. Vite handles this reliably and is already in use in the existing project.
Database
SQLite + SQLCipher
SQLite stores all structured case data in a single file on disk, making it ideal for a local application with no server. SQLCipher is a well-audited open source extension that transparently encrypts every page of the database with AES-256. Without the correct key derived from the master password, the database file is indistinguishable from random noise and is unreadable even with forensic tools.
File Encryption
AES-256-GCM via Node.js crypto
Every uploaded document is encrypted individually before being written to the vault folder. GCM mode provides both confidentiality and integrity verification, meaning any tampering with the ciphertext is detected on open. Decryption happens in memory only and the plain text is never flushed to disk, protecting documents against file system level forensic recovery.
Key Derivation
PBKDF2
Converts the master password into the AES encryption key by running it through a cryptographic hash 100,000 times with a unique random salt. This deliberate computational cost makes brute-force attacks impractically slow: an attacker who obtains the database file would need to run 100,000 hash iterations per password guess rather than computing millions of guesses per second against a plain hash.
PDF Rendering
pdf.js (local build)
Mozilla's pdf.js renders PDF files entirely within the application using JavaScript, with no external service or system PDF viewer involved. This allows Docketly to pass a decrypted document directly from memory into the renderer without it ever touching the file system in plain text. The library is bundled locally at build time, consistent with the zero-network policy.
State Management
Zustand
Manages runtime application state including the in-memory encryption key, the active case, and view state across the timeline, documents, and facts panels. Zustand's minimal footprint and direct store access make it straightforward to implement the security requirement of wiping the key from state on lock without complex side-effect management.
Packaging and Distribution
electron-builder
Produces signed, platform-specific installers: a DMG for macOS and an NSIS installer for Windows. Code signing is essential for distribution to law firms, as unsigned applications trigger operating system security warnings that undermine credibility. electron-builder handles the full signing and packaging pipeline from a single configuration file, targeting both platforms from one codebase.
